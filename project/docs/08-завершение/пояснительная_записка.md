## 1. Введение

### 1.1 Актуальность

В современной строительной отрасли актуальной задачей является автоматизация процессов управления проектами, заказчиками, сметами и исполнителями. Ручной труд приводит к ошибкам, потерям данных и снижению производительности.

### 1.2 Цель проекта

Разработка информационной системы для автоматизации управления строительными проектами, которая позволит:
- Упростить процесс планирования и контроля проектов
- Обеспечить прозрачность взаимодействия между участниками
- Снизить количество ошибок при расчете смет
- Оптимизировать управление ресурсами

### 1.3 Объект и предмет исследования

**Объект:** Процессы управления в строительной компании  
**Предмет:** Информационная система для автоматизации этих процессов

## 2. Анализ предметной области

### 2.1 Основные сущности

**Проект (Project)**
- Описание строительного объекта
- Связи: Заказчик, Смета, Задачи, Договор

**Заказчик (Customer)**
- Информация о клиенте
- Связи: Проекты, Договоры

**Смета (Estimate)**
- Расчет стоимости строительства
- Связи: Проект, Заказчик

**Исполнитель (Employee)**
- Сотрудник компании
- Связи: Задачи

**Договор (Contract)**
- Юридическое соглашение
- Связи: Проект, Заказчик

**Задача (Task)**
- Конкретная работа в проекте
- Связи: Проект, Исполнитель

**Пользователь (User)**
- Аккаунт для входа в систему
- Связи: Все сущности

### 2.2 Основные бизнес-процессы

1. **Создание проекта**
   - Ввод данных о проекте
   - Выбор заказчика
   - Назначение исполнителей

2. **Расчет сметы**
   - Определение стоимости материалов
   - Расчет трудозатрат
   - Итоговая сумма

3. **Управление задачами**
   - Создание задач
   - Назначение исполнителей
   - Контроль сроков

4. **Формирование договоров**
   - Согласование условий
   - Подписание
   - Контроль выполнения

## 3. Архитектура системы

### 3.1 Выбор архитектуры

Для разработки выбрана архитектура **PCMEF** (Presentation → Control → Mediator → Entity → Foundation).

**Преимущества:**
- Четкое разделение ответственностей
- Упрощенное тестирование
- Предотвращение циклических зависимостей
- Легкость внесения изменений

**Структура слоев:**

```
┌─────────────────────────────────────────┐
│         Presentation Layer              │
│   (Web Controllers, Security)           │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│          Control Layer                  │
│      (MVC Controllers)                  │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│         Mediator Layer                  │
│      (Business Logic)                   │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│          Entity Layer                   │
│       (Domain Models)                   │
└───────────────┬─────────────────────────┘
                ↓
┌─────────────────────────────────────────┐
│       Foundation Layer                  │
│   (Repositories, Infrastructure)        │
└─────────────────────────────────────────┘
```

## 4. Проектирование базы данных

### 4.1 Концептуальная модель

**ER-диаграмма:**

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  Project    │◄────►│  Customer   │◄────►│  Estimate   │
└──────┬──────┘      └─────────────┘      └─────────────┘
       │
       │      ┌─────────────┐      ┌─────────────┐
       ├─────►│  Employee   │◄────►│   Task      │
       │      └─────────────┘      └─────────────┘
       │
       │      ┌─────────────┐
       └─────►│  Contract   │
              └─────────────┘
                   ▲
                   │
              ┌─────────────┐
              │   User      │
              └─────────────┘
```

### 4.2 Логическая модель

**Таблицы:**

1. **project**
   - id (PK, BIGSERIAL)
   - name (VARCHAR(255))
   - description (TEXT)
   - start_date (DATE)
   - end_date (DATE)
   - budget (DECIMAL(15,2))
   - status (VARCHAR(50))

2. **customer**
   - id (PK, BIGSERIAL)
   - name (VARCHAR(255))
   - contact_info (TEXT)
   - address (TEXT)

3. **estimate**
   - id (PK, BIGSERIAL)
   - project_id (FK, BIGINT)
   - customer_id (FK, BIGINT)
   - amount (DECIMAL(15,2))
   - date (DATE)

4. **employee**
   - id (PK, BIGSERIAL)
   - name (VARCHAR(255))
   - position (VARCHAR(100))
   - salary (DECIMAL(10,2))
   - phone (VARCHAR(20))
   - email (VARCHAR(100))

5. **contract**
   - id (PK, BIGSERIAL)
   - project_id (FK, BIGINT)
   - customer_id (FK, BIGINT)
   - amount (DECIMAL(15,2))
   - start_date (DATE)
   - end_date (DATE)
   - status (VARCHAR(50))

6. **task**
   - id (PK, BIGSERIAL)
   - project_id (FK, BIGINT)
   - employee_id (FK, BIGINT)
   - description (TEXT)
   - priority (VARCHAR(20))
   - status (VARCHAR(50))
   - due_date (DATE)

7. **users**
   - id (PK, BIGSERIAL)
   - username (VARCHAR(50), UNIQUE)
   - password (VARCHAR(255))
   - role (VARCHAR(20))

### 4.3 Физическая модель

**Индексы:**
- PRIMARY KEY для всех таблиц
- FOREIGN KEY для связей
- UNIQUE INDEX для username в users
- INDEX на frequently queried columns (project_id, customer_id)

### 5.3 Реализация безопасности

**Spring Security настройки:**

```java
@Override
protected void configure(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/login", "/register", "/css/**", "/js/**").permitAll()
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().hasAnyRole("USER", "ADMIN")
        )
        .formLogin(form -> form
            .loginPage("/login")
            .defaultSuccessUrl("/", true)
            .permitAll()
        )
        .logout(logout -> logout
            .logoutUrl("/logout")
            .logoutSuccessUrl("/login?logout")
        );
}
```

