# Описание ORM-стратегии

## Общие принципы

1.  **Использование JPA и Hibernate:** ORM будет реализовано с помощью Java Persistence API (JPA) и ее популярной реализации — Hibernate. Это обеспечит портируемость и соответствует требованиям проекта.
2.  **Аннотации JPA:** Все маппинг-аннотации будут размещены непосредственно в классах сущностей (Entity) пакета `com.skfu.project.entity`.
3.  **Слой Foundation:** Доступ к данным будет осуществляться исключительно через интерфейсы репозиториев в пакете `com.skfu.project.foundation`, которые расширяют `JpaRepository`. Это реализует паттерн Data Mapper.
4.  **Именование:** Имена таблиц и столбцов в БД будут в `snake_case`, а имена классов и полей в Java — в `camelCase`. Связь будет обеспечиваться аннотациями `@Table` и `@Column`.

## Стратегии маппинга

*   **Идентификаторы (ID):** Будет использоваться стратегия `GenerationType.IDENTITY` для автоматической генерации первичных ключей с помощью `BIGSERIAL` в PostgreSQL.
*   **Связи (Relationships):**
    *   **Одна ко Многим (One-to-Many):** Например, `Project` → `Task`. Будет реализована с помощью `@OneToMany` на стороне `Project` и `@ManyToOne` на стороне `Task`. Каскадное удаление (`cascade = CascadeType.ALL`) будет применено для `Project` → `Task`, `Project` → `Estimate`, `Project` → `Contract`, чтобы при удалении проекта удалялись и связанные сущности.
    *   **Один к Одному (One-to-One):** Например, `Project` → `Contract`, `AppUser` → `Employee`. Будет реализована с помощью `@OneToOne`.
    *   **Один ко Многим (One-to-Many) с посредствующей таблицей:** Например, `AppUser` → `Employee`. Свяжем через `employee_id` в таблице `app_user`.

## Управление сессиями и транзакциями

*   **Транзакции:** Управление транзакциями будет осуществляться с помощью аннотации `@Transactional` из Spring на методах Control-слоя. Это гарантирует атомарность операций.
*   **Проблема N+1:** Для предотвращения проблемы N+1 запросов при загрузке связанных сущностей будет использоваться стратегия `JOIN FETCH` в JPQL-запросах или аннотация `@EntityGraph` в репозиториях.

## Безопасность и производительность

*   **Инъекции:** Использование JPA и параметризованных запросов (через Spring Data JPA) полностью исключает возможность SQL-инъекций.
*   **Индексы:** Ключевые поля, используемые в условиях поиска (например, `status` в `Project`, `project_id` в `Task`), будут проиндексированы, как указано в DDL-скриптах.
